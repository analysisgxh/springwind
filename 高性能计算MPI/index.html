    MPI并行计算-Parallel Programming using MPI

    安装配置MPI库

    基本概念 MPI ：

    • MPI概念: 消息传递接口（Message Passing Interface,MPI）
    • CC (Coherence and Cohesion): 连贯与衔接，段落之间的逻辑过渡。
    • MPI的目标: 实现具有高性能和可移植性的大规模并行程序。
    • MPI的介绍: MPI通过在每个节点(机器)启用一个或者多个进程来实现并行计算，每个进程还可以含有多个线程，故有时也将MPI和OpenMP结合使用。MPI

      • 《并行计算导论》张林波等，《并行算法实践》陈国良等。

      VS Code 配置 MS-MPI (C/C++) 环境笔记

    在 VS Code 中配置 MPI 环境，本质上是将 Visual Studio 图形界面里的核心操作（包含目录、库目录、依赖项）通过 JSON 配置文件 或 编译参数 来实现。

    1. 配置代码提示（解决红波浪线）

    对应 VS 操作： 设置 .h 文件目录 (Additional Include Directories)

    • 文件： .vscode/c_cpp_properties.json (或通过 UI 界面设置)
    • 操作：
    • 按 Ctrl + Shift + P，输入并选择 C/C++: Edit Configurations (UI)。
    • 找到 包含路径 (Include path)，添加你的 MPI include 路径。
    • 路径示例： H:/Include（如果是 Intel MPI 则类似 C:/.../mpi/latest/include）。
    • 效果： 告诉 IntelliSense 头文件在哪里，#include <mpi.h> 不再报错。

    2. 配置编译任务（核心步骤）

    对应 VS 操作： 设置 .lib 目录 (Additional Library Directories) * 填入所需的 lib 文件名 (Additional Dependencies)

    • 文件： .vscode/tasks.json
    • 操作： 在 args 数组中添加对应的编译和链接参数。

    方案 A：使用 GCC 编译器 (g++/MinGW)

    "args": [
    "-g",
    "${file}",
    "-o", "${fileDirname}\\${fileBasenameNoExtension}.exe",
    "-I", "H:/Include",                     // 1. 指定头文件目录
    "-L", "H:/Lib/x64",                     // 2. 指定库文件目录
    "-lmsmpi"                               // 3. 链接具体的库 (注意去掉 .lib 后缀)
]

    执行 MPI 程序

    • 程序内不可指定进程数：MPI 程序代码中本身不能指定进程总数，但可以在运行时获取当前环境的总进程数（例如使用 MPI_Comm_size）。
    • 命令行动态指定：在运行该程序时，需要通过命令行参数给定具体的并行规模。
    • 单独执行的局限性：MPI 程序编译完成后，可作为普通程序单独执行（直接运行 .exe），但此时系统仅会在本地启用一个进程（退化为串行程序）。

    • 使用专用启动器：若想在一个或多个节点启用多个并行进程，必须使用专门的程序 mpiexec.exe（或 mpirun、srun）来启动：

      • 基础运行方式: 

        mpiexec [参数] <MPI程序名>

      • 常用参数解析: 

        -n          进程数，缺省为 CPU 核数
-hostfile   后跟文件名字，文件中存储节点名，每行一个
-H, -host   后跟节点名，以逗号分隔
-pernode    每个节点仅启用一个进程

    MPI 运行参数详解

    在分布式计算和高能物理模拟中，合理配置 mpiexec 参数是提升并行效率的关键。以下是常用的运行参数及其使用场景。

    1. -n：指定进程数

    这是最核心的参数。如果不显式指定，MPI 通常会尝试占满当前机器的所有 CPU 逻辑核心。

    • 使用场景：本地调试、单机多核并行。
    • 用法示例：
    本地运行示例
    # 在本地启动 4 个并行进程运行 main.exe
mpiexec -n 4 ./main.exe

    2. -hostfile：从文件读取节点信息

    当计算任务跨越多个服务器（集群）时，通过文件管理节点 IP 更加高效。该文件通常被称为 hosts 或 machinefile。

    • 使用场景：固定节点的集群环境。
    • 配置示例 (my_nodes.txt)：
    my_nodes.txt
    192.168.1.10 slots=2  # 节点 A，可用 2 个核心
192.168.1.11 slots=4  # 节点 B，可用 4 个核心
    • 运行命令：
    读取节点文件运行
    # 总共启动 6 个进程，MPI 会根据 slots 自动分配到不同机器
mpiexec -hostfile my_nodes.txt -n 6 ./main.exe

    3. -H / -host：直接指定运行节点

    如果你只需要临时在特定的几台机器上运行，可以直接在命令行中罗列。

    • 使用场景：临时测试、节点较少的并行任务。
    • 用法示例：
    手动点名运行
    # 在 node1 和 node2 两个节点上各启动一个进程（共计 2 个进程）
mpiexec -host node1,node2 -n 2 ./main.exe

    4. -pernode：每节点单进程模式

    该参数强制每台服务器只启动一个 MPI 进程。

    • 使用场景：混合并行（MPI + OpenMP）。
      • 例如：每个节点运行一个 MPI 进程，然后通过 OpenMP 榨干节点内的所有线程。
    • 用法示例：
    每节点运行一个进程
    # 假设 hostfile 中有 10 台机器，总共会启动 10 个进程
mpiexec -hostfile my_nodes.txt -pernode ./main.exe

    💡 进阶：超算中心环境 (Slurm)

    在大型集群环境下，通常使用 Slurm 作业调度系统。此时不需要手动编写 hostfile，系统会自动根据申请的资源分配节点。

    Slurm 提交建议

    使用 srun 代替 mpiexec 可以更好地与调度器集成，自动处理资源隔离与分配。

    Slurm 任务提交示例
    # 申请 64 个进程，每台机器分配 16 个任务
srun -n 64 --ntasks-per-node=16 ./main.exe

    并行计算核心概念：节点、核心、进程与线程

    在高频振荡的高能物理模拟或大规模 PINN 模型训练中，理清“节点、核心、进程、线程”的关系，是编写出真正能“跑满”超算资源的高效 C++ 代码的前提。

    以下将这四个概念从硬件到软件、从宏观到微观进行解析。

    1. 硬件层面：看得见摸得着的物理实体

    🖥️ 节点 (Node)

    • 定义：超算集群或机房里的一台独立物理计算机（或者虚拟机）。
    • 特征：每个节点有自己独立的操作系统、独立的内存（RAM）和网卡。节点与节点之间必须通过网络线缆（如 InfiniBand 或千兆网）连接。
    • 在 MPI 中的映射：hostfile 里写的每一行 IP 地址，就是一个节点。
    • 通俗比喻：一家拥有独立厂房、独立资金的分公司。

    🧠 核心 (Core)

    • 定义：CPU 内部真正执行数学计算和指令的物理运算单元。
    • 特征：一个 CPU 芯片里可以封装多个核心（比如 16 核、64 核）。它们共享这台节点的物理内存。
    • 在 MPI 中的映射：如果不加限制，程序的最高真实并行度取决于机器的物理核心数。
    • 通俗比喻：分公司里的流水线工位。

    2. 软件层面：操作系统分配任务的虚拟单位

    📦 进程 (Process)

    • 定义：操作系统中一个正在运行的程序的实例。
    • 特征（极其关键）：内存隔离。每个进程都有自己完全独立的内存空间。进程 A 绝对无法直接读取进程 B 的变量。
    • 在 MPI 中的映射：mpiexec -n 4 ./main.exe 就是向系统申请启动 4 个独立的进程。这也是为什么 MPI 需要用 MPI_Send 和 MPI_Recv——因为它们内存不互通，只能通过“发消息”交换物理场数据。
    • 通俗比喻：承包了某个工位的独立外包团队（自带干粮，财务独立，只能打电话沟通）。

    🧵 线程 (Thread)

    • 定义：进程内部更小的执行单元。一个进程可以包含多个线程。
    • 特征（极其关键）：内存共享。同进程内的所有线程，可以直接读写同一片内存里的同一个大矩阵。由于不需要像 MPI 那样走网络发消息，通信极快，但容易产生“数据踩踏”（竞态条件）。
    • 在并行计算中的映射：通常使用 OpenMP 或 C++11 的 std::thread 来控制。
    • 通俗比喻：外包团队里的具体打工人（坐在一张桌子上，共同修改同一份图纸）。

    3. 核心总结与 HPC 黄金组合

    概念 类别 内存状态 通信方式 适用技术栈
    Node (节点) 硬件 节点间物理隔离 网络传输 (慢) -
    Core (核心) 硬件 节点内物理共享 总线传输 (极快) -
    Process (进程) 软件 逻辑隔离 消息传递 (Message Passing) MPI
    Thread (线程) 软件 逻辑共享 直接读写内存 OpenMP / CUDA

    为什么超算里经常用 -pernode 参数？

    因为高能物理计算中最完美的“混合并行架构 (Hybrid Parallelism)”是这样的：

    1. 超算分配给你 10 个节点。
    2. 使用 mpiexec -pernode，在每个节点上只启动 1 个 MPI 进程。
    3. 在 C++ 代码内部，这个 MPI 进程会使用 OpenMP 唤醒 64 个线程。
    4. 这 64 个线程完美占满该节点上的 64 个物理核心，共享内存，极速计算 PDE 残差。
    5. 当需要跨节点交换边界条件时，那 10 个 MPI 进程再通过网络进行通信。

    这就完美避开了“全部用 MPI 导致节点内内存重复占用”和“全部用 OpenMP 导致跨不出单台机器”的死局。
